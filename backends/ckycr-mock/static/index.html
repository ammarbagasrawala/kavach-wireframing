CKYC registry mock — flow UI

Browser UI with autofill presets. All fields stay editable. Secured Search uses server-side keys to build signed XML (same as the Python example script).

End-to-end flow (hosting)

  1. Keys on the server — Keep keys/cersai_private.pem, keys/cersai_public.pem, keys/fi_private.pem, keys/fi_public.pem on the host filesystem (or map env paths in app/config.py). Never commit private keys. Register the FI’s public PEM in the database (step 2).
  2. Register FI — One row in fi_institution with fi_code + fi_public_key_pem so verify can check XML-DSig and encrypt the response session key.
  3. Batch upload (optional) — JSON to /ckyc/batch/upload fills kyc_record (PAN, photo, etc.).
  4. Secured search — Production clients send signed XML to /Search/ckycverificationservice/verify. This UI calls /api/demo/verify-search which builds that XML on the server using the same crypto — for demos only.

4. Secured search (demo)

        

          Uses server keys to build v1.3 XML and runs verify in-process. The demo JSON body only has
          id_type and id_no — there is no separate PAN key. For PAN search,
          use type C and put the PAN in id_no; inside the encrypted PID that becomes
          <ID_TYPE>C</ID_TYPE> and <ID_NO>…</ID_NO>.
          C = PAN (4th char ABCFGHJLPT);
          E = Aadhaar-style last4|name|DD-MM-YYYY|gender;
          Z / Y = CKYC number / reference;
          A,B,D,F,G,02,03 = matched via id_json on the record (see seed).
          v1.3 search PID_DATA response does not define a <PAN> element.
          Z/Y fixed presets match scripts/seed.py only; after batch upload, copy
          CKYC numbers from batch status. E must match stored aadhaar last four + name + DOB + gender
          (default batch JSON includes them).
        

        
        

        
        

          

            
            
            
          

          
          
        

        

          Response fields: http_status — HTTP result;
          request_id_used — idempotency key for the day;
          request_xml — signed REQ_ROOT you effectively sent;
          response_xml — CERSAI-signed reply (encrypted PID + session key);
          decrypted_pid_xml — KYC payload after decrypting with the FI private key (what a real FI client would derive).
          PAN / Aadhaar: v1.3 response PID_DATA does not include a PAN tag; full Aadhaar is not returned. Type E search uses last four + demographics only.
